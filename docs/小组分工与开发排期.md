# PDP-System 小组分工与开发排期说明

## 1. 文档目的

本文档用于明确当前项目尚未完成的功能缺口，并结合已确定的小组分工，细化每位成员的具体开发内容、实施方式、协作依赖和阶段性交付目标。同时给出按周推进的开发顺序，便于全组并行开发和后续联调。

## 2. 当前项目实现现状

结合当前仓库代码，项目整体处于“前端原型已较完整、部分后端接口已具备、但多个模块尚未形成前后端闭环”的状态。

### 2.1 已具备基础的部分

1. 用户登录/注册已有基本实现。
2. 教师端已有基础入口和部分学生列表相关接口。
3. 课程、成就、简历、对比分析都已有前端页面雏形。
4. 后端已经存在课程、成就、文件、学生、教师等基础路由。

### 2.2 目前仍未完整实现的部分

1. 登录注册基础已实现，但细粒度权限控制还不完整，可参考 app/api/v1/auth.py 和 frontend/modules/composables/useAuth.js。
2. 课程模块前端页面较完整，但前端目前主要还是本地数据操作，后端只有查询和新增，缺编辑、删除和完整联调，可参考 frontend/modules/composables/useCourses.js 和 app/api/v1/courses.py。
3. 成就模块后端 CRUD 基本有了，但前端还没真正接后端，附件上传也没闭环，可参考 frontend/modules/composables/useAchievements.js、app/api/v1/achievements.py、app/api/v1/files.py。
4. 简历模块现在基本是前端模板和假导出，没有真实导出链路，也没有后端支持，可参考 frontend/modules/composables/useResume.js。
5. 大众对比模块现在基本是页面原型和前端写死结果，没有真正的数据导入、清洗、分析、后端计算接口，可参考 frontend/modules/composables/useCompare.js。
6. 教师端已有入口和部分后端接口，但教师看到的很多学生画像数据还是占位数据，不是真实从课程和成就聚合来的，可参考 frontend/modules/composables/useTeacher.js 和 app/api/v1/teachers.py。

## 3. 小组分工总览

### 3.1 成员分工

1. 李晔：简历生成与导出
2. 王颖超：课程记录、成绩管理与 GPA 计算

### 3.2 分工原则

2. 每个模块都需要负责本模块的页面、接口联调、数据结构整理和基础验证。
3. 公共能力如权限控制、统一数据字段、联调规则由全组共同约定，默认由组长统筹推进。

## 4. 各成员详细任务说明

## 4.1 李晔：简历生成与导出

### 负责目标

实现学生简历模块的完整功能，使系统能够基于真实学生数据生成简历，支持预览和导出。

### 当前缺口

1. 简历信息当前大多为前端写死数据。
2. 简历模块尚未完全接入课程和成就模块的真实数据。
3. 导出功能目前仅为前端提示，没有真实文件输出。

### 你需要完成的内容

1. 完善学生基本信息维护功能，包括姓名、电话、邮箱、学校、专业、求职方向、技能标签、照片等。
2. 将课程模块与成就模块的真实数据接入简历展示。
3. 完善简历模板切换和预览功能，至少保证单栏模板和双栏模板可正常使用。
4. 实现 PDF 导出功能，条件允许时再补充 Word 导出。
5. 增加简历完整度提示和缺失字段提示。

### 具体实施方式

1. 先改 frontend/modules/composables/useResume.js，把现在写死的个人信息改成可编辑、可保存的数据结构。
2. 和课程负责人、成就负责人约定数据格式：
课程至少要给你课程名、成绩、学分、GPA。
成就至少要给你项目、实习、竞赛、证书、奖项这些分类后的结构。
3. 在前端先完成“预览正确渲染”，确保简历页面展示无问题。
4. 导出实现建议先做前端导出：
第一阶段先做浏览器打印导出 PDF。
第二阶段再考虑 Word 导出，或者后端生成文件。
5. 如果时间够，可以新增后端简历导出接口；如果时间不够，就先完成前端 PDF 导出，Word 作为次优目标。

### 对外依赖

1. 依赖王颖超提供课程真实数据。
2. 依赖薛军浩提供成就分类数据。

### 阶段性交付

1. 简历预览页可用。
2. 能根据课程和成就自动生成简历内容。
3. 至少支持 PDF 导出。
4. 缺失字段时有明确提示。
5. 单栏/双栏模板可切换。

## 4.2 王颖超：课程记录、成绩管理与 GPA 计算

### 负责目标

实现课程模块的前后端完整闭环，使学生能够录入、编辑、删除课程，并自动完成 GPA 及相关统计计算。

### 当前缺口

1. 前端课程数据目前主要为本地模拟数据。
2. 后端仅支持课程列表和新增，缺少编辑与删除接口。
3. 教师端需要的课程摘要数据尚未统一提供。

### 你需要完成的内容

1. 实现课程列表查询、课程新增、课程编辑、课程删除。
2. 统一 GPA 计算规则，并在学生端稳定展示。
3. 将课程模块从本地数据改为真实调用后端接口。
4. 为教师端提供课程摘要字段，如 GPA、总课程数、核心课程等。
5. 做基础测试或手动验证用例。
6. 补齐后端缺少的更新、删除接口。

### 具体实施方式

1. 先看 app/api/v1/courses.py，现在只有列表和新增。
2. 你需要补：
PUT /api/v1/students/{student_id}/courses/{course_id}
DELETE /api/v1/students/{student_id}/courses/{course_id}
3. 再改 frontend/modules/composables/useCourses.js，把 initCourses、saveCourse、deleteCourse 从本地数据改成 API 调用。
4. 课程编辑时，前端表单要能回填，保存后要刷新列表。
5. GPA 计算要统一规则，不要教师端一套、学生端一套。你要把规则写清楚，作为全组统一标准。
6. 给教师端输出聚合结果，至少包括：
平均 GPA
总课程数
高分课程
核心课程概览

### 对外依赖

1. 你的模块是简历模块、教师端模块、对比分析模块的重要数据来源。

### 阶段性交付

1. 课程 CRUD 完整可用。
2. GPA 计算准确。
3. 页面刷新后数据不丢失。
4. 教师端可读取课程摘要信息。
5. 至少补一组课程相关测试或手动测试用例。

## 4.4 魏吾嘉：大众数据导入与对比分析

### 负责目标

实现基准数据导入、数据清洗、多维度对比分析和统一结果输出，使学生端和教师端都能基于真实数据得到分析结果。

### 当前缺口

1. 当前对比分析结果为前端写死内容。
2. 没有真实的基准数据导入和校验流程。
3. 缺少后端分析接口与统一返回结构。

### 你需要完成的内容

1. 设计并确定基准数据导入格式，如 Excel 或 CSV 的字段要求。
2. 实现导入校验、错误提示、数据清洗和结果反馈。
3. 完成对比分析逻辑，不再使用前端写死结论。实现至少 3 到 4 个核心维度的对比分析。
4. 输出学生端和教师端可共用的统一分析结果结构。
5. 支持图表展示所需的数据格式。

### 具体实施方式

1. 先看 frontend/modules/composables/useCompare.js，这里现在只是前端假数据。
2. 你需要优先补后端分析接口，建议新增一组接口，例如：
基准数据导入接口
对比分析执行接口
对比结果查询接口
3. 导入时至少要校验：
学号格式
GPA 是否为空
年级/专业/学校字段是否完整
成就类字段是否缺失
4. 先只做最核心的几个对比维度，不要一开始铺太大：
GPA
课程修读情况
竞赛/奖项数量
实习/项目数量
5. 输出结果时要给前端统一格式，比如：
总结论
每个维度的个人值
群体均值/排名区间
图表数据
6. 你的结果要能同时支持学生端和教师端，所以不要把展示逻辑写死在一个页面里。

### 对外依赖

1. 依赖课程和成就模块提供稳定字段。
2. 输出结果需要被学生端和教师端共同复用。

### 阶段性交付

1. 基准数据可导入。
2. 导入失败可提示原因。
3. 对比分析结果不再是写死数据。
4. 学生端和教师端都可使用统一结果。
5. 至少支持 3 到 4 个核心维度。

## 7. 建议的验收顺序

建议项目验收按照以下顺序进行：

1. 学生登录后录入课程。
2. 学生录入成就并上传附件。
3. 学生生成并预览简历，导出 PDF。
4. 学生导入或使用基准数据执行对比分析。
5. 教师登录后查看自己的学生列表。
6. 教师查看学生画像和对比结论。

## 8. 组内协作建议

1. 每周至少进行一次字段和接口同步，避免模块之间的数据格式不一致。
2. 每个模块在开发前先写清楚输入和输出结构，再开始写页面和接口。
3. 先保证功能闭环，再做页面美化，避免时间后期不足。
4. 联调时优先解决接口可用性和数据一致性问题，不要先纠结样式细节。

## 9. 结论

当前项目的关键不是继续扩展新功能，而是把已有的课程、成就、简历、对比分析、教师端五大模块真正打通，形成完整闭环。按照本文档的人员分工和每周推进顺序执行，可以较稳地完成项目实现、联调和最终演示。
